# Real MySQL 9.1 ~9.2 정리

  

## 1. 개요

  

### 실행 순서

  

MySQL에서 쿼리가 실행되는 과정은 크게 다음 3단계로 나눌 수 있다.

  

1. SQL 문장을 잘게 쪼개 MySQL 서버가 이해할 수 있는 수준의 파스 트리로 만든다.

2. 파싱 정보를 확인하면서 어떤 테이블을 읽을지, 어떤 인덱스를 사용할지 선택한다.

3. 결정된 테이블 읽기 순서와 선택된 인덱스를 이용해 스토리지 엔진으로부터 데이터를 가져온다.

  

1단계는 **SQL 파싱**이라고 하며, SQL 파서 모듈이 처리한다. 문법적으로 잘못된 SQL은 이 단계에서 걸러진다. 이때 SQL 파스 트리가 만들어지고, MySQL은 이 파스 트리를 기반으로 SQL을 실행한다.

  

2단계는 **최적화 및 실행 계획 수립** 단계이며, 옵티마이저가 처리한다. 이 단계가 완료되면 실행 계획이 만들어진다.

  

3단계에서는 스토리지 엔진에서 레코드를 읽어오고, 필요에 따라 조인이나 정렬 작업도 수행한다.

  

1, 2단계는 주로 MySQL 엔진에서 처리하며, 3단계는 MySQL 엔진과 스토리지 엔진이 함께 참여해서 처리한다.

  

### 옵티마이저 종류

  

옵티마이저는 크게 **비용 기반 최적화(CBO)** 와 **규칙 기반 최적화(RBO)** 로 나눌 수 있다.

  

**비용 기반 최적화(CBO, Cost Based Optimizer)** 는 쿼리를 처리하기 위한 여러 가지 가능한 방법을 만들고, 각 단위 작업의 비용을 계산한 뒤 비용이 가장 적게 드는 방법을 선택하는 방식이다. 대부분의 DBMS가 이 방식을 사용한다.

  

**규칙 기반 최적화(RBO, Rule Based Optimizer)** 는 옵티마이저에 내장된 우선순위 규칙에 따라 실행 계획을 수립하는 방식이다.

  

## 실습 준비

아래 실습은 MySQL 8.0 기준으로 실행해볼 수 있다. 각 개념을 읽은 뒤 바로 `EXPLAIN`이나 `EXPLAIN ANALYZE`를 붙여 실행 계획을 확인하면 된다.

먼저 실습용 데이터베이스와 테이블을 만든다.

```sql
DROP DATABASE IF EXISTS optimizer_lab;
CREATE DATABASE optimizer_lab;
USE optimizer_lab;

CREATE TABLE member (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    age INT NOT NULL,
    joined_at DATETIME NOT NULL,
    region VARCHAR(20) NOT NULL,
    bio TEXT,
    INDEX idx_member_age_joined_at (age, joined_at),
    INDEX idx_member_joined_at (joined_at),
    INDEX idx_member_region_age (region, age)
) ENGINE=InnoDB;

CREATE TABLE reservation (
    id BIGINT PRIMARY KEY,
    member_id BIGINT NOT NULL,
    theme_id BIGINT NOT NULL,
    reservation_time DATETIME NOT NULL,
    status VARCHAR(20) NOT NULL,
    INDEX idx_reservation_member_id (member_id),
    INDEX idx_reservation_time (reservation_time),
    INDEX idx_reservation_member_time (member_id, reservation_time)
) ENGINE=InnoDB;
```

실습 데이터를 넣는다.

```sql
SET cte_max_recursion_depth = 10000;

INSERT INTO member (id, name, age, joined_at, region, bio)
WITH RECURSIVE seq AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1
    FROM seq
    WHERE n < 1000
)
SELECT
    n,
    CONCAT('member-', n),
    20 + (n % 40),
    DATE_ADD('2024-01-01', INTERVAL n DAY),
    ELT((n % 5) + 1, 'SEOUL', 'BUSAN', 'DAEGU', 'INCHEON', 'GWANGJU'),
    RPAD(CONCAT('bio-', n), 500, '*')
FROM seq;

INSERT INTO reservation (id, member_id, theme_id, reservation_time, status)
WITH RECURSIVE seq AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1
    FROM seq
    WHERE n < 10000
)
SELECT
    n,
    ((n - 1) % 1000) + 1,
    (n % 30) + 1,
    DATE_ADD('2024-01-01', INTERVAL n HOUR),
    IF(n % 3 = 0, 'CANCELED', 'RESERVED')
FROM seq;
```

## 2. 기본 데이터 처리

  

### 풀 테이블 스캔

  

풀 테이블 스캔은 인덱스를 사용하지 않고 테이블의 데이터를 처음부터 끝까지 읽어서 요청된 작업을 처리하는 방식이다.

  

주로 다음과 같은 경우 선택된다.

  

- 테이블의 레코드 건수가 너무 작은 경우

- 쿼리에 적절한 조건이 없는 경우

- 조건에 일치하는 레코드 건수가 너무 많은 경우

  

InnoDB에서는 포그라운드 스레드가 읽기를 시작하다가, 특정 시점부터 백그라운드 스레드가 미리 데이터를 읽는 **리드 어헤드(Read Ahead)** 작업을 수행한다.

실습으로 확인해보자. 인덱스를 사용하기 어려운 조건을 걸면 풀 테이블 스캔이 발생할 수 있다.

```sql
EXPLAIN
SELECT *
FROM member
WHERE bio LIKE '%777%';
```

### 풀 인덱스 스캔

  

풀 인덱스 스캔은 인덱스를 처음부터 끝까지 스캔하는 방식이다.

  

예를 들어 다음과 같이 전체 레코드 건수를 조회하는 경우, 테이블보다 크기가 작은 인덱스를 읽는 편이 디스크 읽기 횟수를 줄일 수 있다.

  

```sql

```

### 병렬 처리

  

MySQL에서는 다음 설정으로 병렬 읽기 스레드 수를 조절할 수 있다.

  

```sql

SET SESSION innodb_parallel_read_threads = {num};

```

  

다만 병렬 처리는 `WHERE` 조건 없이 단순하게 테이블 전체 건수를 가져오는 쿼리에서만 사용할 수 있다.

  

병렬 처리용 스레드 개수를 무작정 늘린다고 항상 성능이 좋아지는 것은 아니다. 서버의 CPU 코어 개수를 넘어서는 수준으로 스레드를 늘리면 오히려 성능이 떨어질 수 있다.

  

## 3. ORDER BY 처리

  

### 정렬 처리 방식

  

MySQL에서 `ORDER BY`를 처리하는 방식은 크게 두 가지다.

  

#### 인덱스를 이용한 정렬

  

인덱스는 이미 정렬된 상태로 관리되므로, 인덱스 순서대로 읽기만 하면 된다. 따라서 매우 빠르다.

  

다만 레코드 추가나 변경 시 인덱스도 함께 갱신해야 하므로 쓰기 작업은 느려질 수 있고, 디스크와 메모리 공간도 더 필요하다.

  

#### Filesort를 이용한 정렬

  

인덱스를 사용할 수 없는 경우 MySQL은 별도의 정렬 작업을 수행한다. 이를 Filesort라고 한다.

  

정렬해야 할 레코드가 많지 않다면 메모리에서 빠르게 처리할 수 있지만, 레코드가 많아질수록 성능이 저하될 수 있다.

  

실행 계획의 `Extra` 컬럼에 `Using filesort`가 표시되면 Filesort를 사용한 것이다.

인덱스를 이용한 정렬과 Filesort를 비교해보자.

```sql
EXPLAIN
SELECT id, age, joined_at
FROM member
ORDER BY age, joined_at
LIMIT 10;
```

![](<image/Pasted image 20260518171322.png>)
확인할 것:

- `key`가 `idx_member_age_joined_at`인지 확인한다.
- `Extra`에 `Using filesort`가 없으면 인덱스를 이용해 정렬한 것이다.

반대로 인덱스 컬럼 순서와 다르게 정렬하면 Filesort가 발생할 수 있다.

```sql
EXPLAIN
SELECT id, age, joined_at
FROM member
ORDER BY joined_at, age
LIMIT 10;
```

확인할 것:

- `Extra`에 `Using filesort`가 표시되는지 확인한다.

  

### 모든 정렬에 인덱스를 사용할 수 없는 이유

  

모든 정렬 조건에 맞춰 인덱스를 만드는 것은 현실적으로 어렵다.

  

대표적인 이유는 다음과 같다.

  

- 정렬 기준이 너무 많아 모든 요건별로 인덱스를 만들 수 없는 경우

- `GROUP BY` 결과나 `DISTINCT` 처리 결과를 정렬해야 하는 경우

- `UNION` 결과처럼 임시 테이블의 결과를 다시 정렬해야 하는 경우

- 랜덤하게 결과 레코드를 가져와야 하는 경우

  

## 4. 소트 버퍼

  

MySQL은 정렬을 수행하기 위해 별도의 메모리 공간을 할당받아 사용한다. 이를 **소트 버퍼(Sort Buffer)** 라고 한다.

  

정렬할 레코드가 소량이면 소트 버퍼 안에서 처리할 수 있다.

  

하지만 레코드가 많아 소트 버퍼에 한 번에 담을 수 없다면, MySQL은 정렬할 데이터를 여러 번 나누어 읽고 정렬한 뒤 그 결과를 임시로 디스크에 저장한다. 이후 정렬된 결과들을 다시 병합하면서 최종 정렬 결과를 만든다.

  

이 병합 작업을 **멀티 머지(Multi Merge)** 라고 한다.

  

## 5. 정렬 알고리즘

  

MySQL의 정렬 알고리즘은 정렬할 때 어떤 데이터를 소트 버퍼에 담는지에 따라 나눌 수 있다.

```sql
FROM employees
ORDER BY first_name
```

정렬 후 실제 SELECT 대상 컬럼을 가져오기 위해 테이블을 다시 읽어야 한다. 따라서 테이블을 두 번 읽어야 하는 단점이 있다.

## 6. ORDER BY의 세 가지 처리 방법

  

쿼리에 `ORDER BY`가 사용되면 MySQL은 다음 세 가지 방법 중 하나로 정렬을 처리한다. 아래로 갈수록 처리 속도가 느려진다.

  

| 처리 방법 | 실행 계획 Extra |

| --- | --- |

| 인덱스를 사용한 정렬 | 별도 표기 없음 |

| 조인에서 드라이빙 테이블만 정렬 | `Using filesort` |

| 조인 결과를 임시 테이블로 저장 후 정렬 | `Using temporary; Using filesort` |


다음 쿼리를 기준으로 각 정렬 처리 방법의 비용을 비교해보자.

```sql
SELECT *
FROM tb_test1 t1, tb_test2 t2
WHERE t1.col1 = t2.col1
ORDER BY t1.col2
LIMIT 10;
```

가정은 다음과 같다.

- `tb_test1` 테이블의 레코드 수는 100건이다.
- `tb_test2` 테이블의 레코드 수는 1,000건이다.
- `tb_test1`의 레코드 1건당 `tb_test2`의 레코드가 10건씩 매칭된다.
- 전체 조인 결과는 1,000건이다.
- 정렬 기준은 `t1.col2`이고, 최종 결과는 `LIMIT 10`으로 10건만 필요하다.

| 정렬 처리 방법 | 읽어야 할 건수 | 조인 횟수 | 정렬해야 할 대상 건수 |
| --- | --- | --- | --- |
| 인덱스 사용 | `tb_test1`: 1건<br>`tb_test2`: 10건 | 1번 | 0건 |
| 조인의 드라이빙 테이블만 정렬 | `tb_test1`: 100건<br>`tb_test2`: 10건 | 1번 | 100건 |
| 임시 테이블 사용 후 정렬 | `tb_test1`: 100건<br>`tb_test2`: 1,000건 | 100번 | 1,000건 |

인덱스를 사용하면 `t1.col2` 순서대로 `tb_test1`을 읽을 수 있으므로, 첫 번째 `tb_test1` 레코드와 매칭되는 `tb_test2` 10건만 찾으면 `LIMIT 10`이 채워진다. 따라서 별도 정렬 대상은 없다.

조인의 드라이빙 테이블만 정렬하는 경우에는 `tb_test1` 100건을 먼저 정렬한 뒤, 정렬된 첫 번째 `tb_test1` 레코드와 매칭되는 `tb_test2` 10건만 찾으면 된다.

임시 테이블을 사용하는 경우에는 조인 결과 1,000건을 모두 만든 뒤 정렬해야 한다. `LIMIT 10`이 있어도 정렬 전에는 어떤 10건이 앞에 올지 알 수 없기 때문에 가장 비용이 크다.


### 1. 인덱스를 사용한 정렬

  

인덱스를 사용한 정렬이 가능하려면 `ORDER BY`에 명시된 컬럼이 가장 먼저 읽는 테이블에 속해야 하며, `ORDER BY` 순서대로 생성된 인덱스가 있어야 한다.

```
CREATE TABLE member (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    created_at DATETIME
);

CREATE INDEX idx_member_age_created_at
ON member (age, created_at);
```
  
라는 테이블이 존재할 떄,

```
SELECT *
FROM member
ORDER BY age, created_at;
```
이건 가능하다. 왜냐면 age, creaated_at 순서로 정렬되어있기 때문이다.
반대로는 안되겠죠?


또한 
```
CREATE TABLE reservation (
    id BIGINT PRIMARY KEY,
    member_id BIGINT,
    reservation_time DATETIME,
    theme_id BIGINT
);

CREATE INDEX idx_reservation_time
ON reservation (reservation_time);
```
의 상황에서

```
SELECT *
FROM reservation r
JOIN member m ON r.member_id = m.id
ORDER BY r.reservation_time;
```
는 작동한다.왜냐면 맨 처음 읽는 테이블 (드라이빙 테이블)에 인덱스가 되어있기 떄문.


해시 인덱스나 전문 검색 인덱스는 정렬에 사용할 수 없다.

  

여러 테이블이 조인되는 경우에는 네스티드 루프 방식의 조인에서만 이 방식을 사용할 수 있다.
네스티드 루프는 뭐냐?
```
네스티드 루프 조인은 말 그대로 **반복문 안에 반복문이 들어간 방식의 조인**이에요.

예를 들어 이런 쿼리가 있다고 해볼게요.

`SELECT * FROM reservation r JOIN member m ON r.member_id = m.id;`

MySQL이 reservation을 먼저 읽는다고 하면 동작은 대략 이렇습니다.

`for (reservation r : reservations) { 
    member m = findMemberById(r.member_id); 
if (m != null) { 
    result.add(r, m); } }`

즉:

1. 먼저 한 테이블에서 레코드 하나를 읽는다.
2. 그 레코드와 매칭되는 데이터를 다른 테이블에서 찾는다.
3. 이 과정을 첫 번째 테이블의 레코드 수만큼 반복한다.

여기서 먼저 읽는 테이블을 **드라이빙 테이블**이라고 하고, 나중에 매칭해서 읽는 테이블을 **드리븐 테이블**이라고 해요.

`reservation r -> member m 드라이빙 테이블 드리븐 테이블`

예를 들어 reservation이 100건이고, 각 예약의 member_id로 member를 찾는다면:

`reservation 1건 읽음 -> member에서 매칭되는 1건 찾음 reservation 1건 읽음 -> member에서 매칭되는 1건 찾음 reservation 1건 읽음 -> member에서 매칭되는 1건 찾음 ... 100번 반복`

그래서 이름이 **Nested Loop**, 중첩 반복문입니다.

중요한 포인트는 드리븐 테이블 쪽 조인 컬럼에 인덱스가 있으면 빠르다는 거예요.

`CREATE INDEX idx_member_id ON member(id);`

그러면 매번 member 전체를 뒤지는 게 아니라, 인덱스로 바로 찾아갈 수 있습니다.

`좋은 경우: reservation 1건 읽기 -> member.id 인덱스로 바로 찾기 나쁜 경우: reservation 1건 읽기 -> member 전체 스캔`

정리하면:

네스티드 루프 조인은 한쪽 테이블을 한 건씩 읽으면서, 그때마다 다른 테이블에서 매칭되는 행을 찾아오는 조인 방식입니다. 반복문 안에 반복문이 있는 구조라서 이렇게 부르고, MySQL에서 가장 기본적으로 많이 쓰이는 조인 방식이에요.
```

### 2. 조인의 드라이빙 테이블만 정렬

  

일반적으로 조인이 수행되면 결과 레코드 건수와 레코드 크기가 커진다. (당연)
따라서 조인을 실행하기 전에 첫 번째로 읽는 테이블, 즉 드라이빙 테이블의 레코드만 먼저 정렬한 뒤 조인을 수행하는 것이 차선책이 될 수 있다.

  
이 방식으로 처리되려면 ORDER BY 절이 첫 번째로 읽히는 테이블의 컬럼만으로 작성되어야 한다.

예를 들어 다음과 같은 테이블이 있다고 가정한다.

```
CREATE TABLE member ( 
id BIGINT PRIMARY KEY, 
name VARCHAR(50), 
joined_at DATETIME );
 
CREATE TABLE reservation ( 
id BIGINT PRIMARY KEY, 
member_id BIGINT, 
reservation_time DATETIME, 
theme_id BIGINT, 
INDEX idx_reservation_member_id (member_id) );
```

그리고 다음 쿼리를 실행한다고 해보자.

`SELECT * FROM member m JOIN reservation r ON r.member_id = m.id ORDER BY m.joined_at LIMIT 10;`

이 쿼리에서 ORDER BY 기준 컬럼인 m.joined_at은 첫 번째로 읽는 테이블인 member에 속한다. 따라서 MySQL은 조인 결과 전체를 만든 뒤 정렬하지 않고, 먼저 member 테이블만 joined_at 기준으로 정렬한 다음 조인을 수행할 수 있다.

처리 순서는 대략 다음과 같다.

1. member 테이블에서 조건에 맞는 레코드를 읽는다.
2. 읽은 member 레코드들을 m.joined_at 기준으로 정렬한다.
3. 정렬된 member 레코드를 앞에서부터 하나씩 읽는다.
4. 각 member.id 값으로 reservation.member_id 인덱스를 이용해 매칭되는 예약을 찾는다.
5. 조인 결과를 만들어 클라이언트에 반환한다.
6. LIMIT 10을 채우면 더 이상 읽지 않고 멈출 수 있다.

즉 개념적으로는 다음과 비슷하다.

```
List<Member> members = readMembers(); 
members.sort(byJoinedAt);
for (Member member : members) { 
	List<Reservation> reservations = findReservationsByMemberId(member.id);
for (Reservation reservation : reservations) { 
	result.add(member, reservation); 
if (result.size() == 10) { 
	return result; } } }
```

이 방식의 장점은 정렬 대상이 조인 결과 전체가 아니라 드라이빙 테이블인 member의 레코드로 제한된다는 점이다.

예를 들어 member가 100건이고, 회원 한 명당 reservation이 10건씩 있다면 조인 결과는 1,000건이다.

하지만 이 방식에서는 조인 결과 1,000건을 모두 정렬하지 않고, 먼저 member 100건만 정렬한 뒤 조인을 수행할 수 있다.

`정렬 대상: member 100건 조인 결과 전체: member 100건 * reservation 10건 = 1,000건`

다만 이 방식은 ORDER BY가 드라이빙 테이블의 컬럼만으로 구성되어 있을 때 가능하다. 만약 다음처럼 드리븐 테이블인 reservation의 컬럼으로 정렬한다면, member만 먼저 정렬해서는 최종 결과 순서를 보장할 수 없다.

`SELECT * FROM member m JOIN reservation r ON r.member_id = m.id ORDER BY r.reservation_time LIMIT 10;`

이 경우에는 조인 결과를 만든 뒤 r.reservation_time 기준으로 정렬해야 하므로 임시 테이블을 이용한 정렬이 필요할 가능성이 높다.

실습할 때는 옵티마이저가 테이블 순서를 바꿔버릴 수 있으므로 `STRAIGHT_JOIN`을 사용해서 `member`를 먼저 읽도록 고정하면 이해하기 쉽다.

```sql
EXPLAIN
SELECT m.id, m.name, r.id AS reservation_id
FROM member m
STRAIGHT_JOIN reservation r ON r.member_id = m.id
ORDER BY m.name
LIMIT 10;
```

확인할 것:
![](<IMAGE/Pasted image 20260518173015.png>)

- `Extra`에 `Using filesort`가 표시될 수 있다.
- `Using temporary`가 없다면 조인 결과 전체를 임시 테이블에 담아 정렬하는 방식은 아니다.
- `m.name`에는 인덱스를 만들지 않았기 때문에 인덱스 정렬이 아니라 Filesort를 확인하기 좋다.

  

### 3. 임시 테이블을 이용한 정렬

  

2개 이상의 테이블을 조인한 결과를 정렬해야 하는 경우 임시 테이블이 필요할 수 있다.

  

이 방식은 조인 결과를 임시 테이블에 저장한 뒤, 그 결과를 다시 정렬한다.

  

`ORDER BY` 기준 컬럼이 드리븐 테이블에 있는 경우 이 방식을 사용할 가능성이 높다.

실습으로 확인해보자. 아래 예시는 `member`를 먼저 읽도록 고정했지만, 정렬 기준은 드리븐 테이블인 `reservation`에 있다.

```sql
EXPLAIN
SELECT m.id, m.joined_at, r.id AS reservation_id, r.reservation_time
FROM member m
STRAIGHT_JOIN reservation r ON r.member_id = m.id
ORDER BY r.reservation_time
LIMIT 10;
```

확인할 것:

- `Extra`에 `Using temporary; Using filesort`가 표시될 수 있다.
- 조인 결과를 만든 뒤 `r.reservation_time` 기준으로 다시 정렬해야 하기 때문이다.

  

## 7. 쿼리 처리 방식 비교

  

### 스트리밍 방식

  

스트리밍 방식은 서버에서 처리할 데이터가 얼마나 많은지와 관계없이, 조건에 일치하는 레코드가 검색될 때마다 클라이언트로 바로 전송하는 방식이다.

  

첫 응답 속도가 빠르다는 장점이 있다. 다만 마지막 레코드가 언제 전달될지는 예측하기 어렵다.

  

`LIMIT`처럼 결과 건수를 제한하는 조건이 있으면 전체 쿼리 실행 시간을 줄일 수 있다.

  

웹 서비스 같은 OLTP 환경에서는 응답 시간이 중요하므로 스트리밍 방식이 유리하다.

  

`ORDER BY` 처리 방식 중에서는 **인덱스를 이용한 정렬 방식만 스트리밍 형태로 처리**된다. 나머지 방식은 모두 버퍼링되어 정렬된다.

인덱스를 이용한 정렬은 앞에서부터 읽다가 `LIMIT` 개수를 채우면 멈출 수 있다.

```sql
EXPLAIN ANALYZE
SELECT id, age, joined_at
FROM member
ORDER BY age, joined_at
LIMIT 10;
```

반면 Filesort 방식은 상위 10건만 필요하더라도 정렬을 위해 더 많은 레코드를 읽어야 할 수 있다.

```sql
EXPLAIN ANALYZE
SELECT id, age, joined_at
FROM member
ORDER BY name
LIMIT 10;
```

확인할 것:

- 인덱스 정렬은 `LIMIT 10`의 효과를 크게 볼 수 있다.
- Filesort는 결과 10건만 반환해도 정렬을 위한 선행 작업이 필요하다.



### 정렬 관련 상태 변수

MySQL에서 정렬이 실제로 얼마나 발생했는지는 다음 명령으로 확인할 수 있다.

```sql
SHOW SESSION STATUS LIKE 'Sort%';
```

또는 현재 세션이 아니라 서버 전체 누적 값을 보고 싶다면 다음처럼 확인한다.

```sql
SHOW GLOBAL STATUS LIKE 'Sort%';
```

주요 상태 변수는 다음과 같다.

| 상태 변수 | 의미 |
| --- | --- |
| `Sort_scan` | 테이블 스캔을 통해 정렬이 수행된 횟수 |
| `Sort_range` | 인덱스 range scan 결과에 대해 정렬이 수행된 횟수 |
| `Sort_rows` | 정렬된 전체 레코드 수 |
| `Sort_merge_passes` | 소트 버퍼에 담지 못해 디스크 임시 파일을 병합한 횟수 |

실습할 때는 쿼리 실행 전후로 값을 비교하면 된다.

```sql
SHOW SESSION STATUS LIKE 'Sort%';

SELECT id, age, joined_at
FROM member
ORDER BY name
LIMIT 10;

SHOW SESSION STATUS LIKE 'Sort%';
```
![alt text](<image/Pasted image 20260518173147.png>)
`Sort_rows`가 증가했다면 실제 정렬 대상 레코드가 있었다는 뜻이다. `Sort_merge_passes`가 증가했다면 소트 버퍼만으로 정렬을 끝내지 못하고 디스크 임시 파일 병합까지 발생했다는 뜻이므로, 정렬 비용이 커졌다고 볼 수 있다.

반대로 인덱스를 이용한 정렬은 이미 정렬된 인덱스를 순서대로 읽기만 하므로, Filesort 관련 상태 변수 증가가 없거나 훨씬 적을 수 있다.

### 버퍼링 방식

  

버퍼링 방식은 모든 레코드를 가져온 뒤 정렬하거나 그루핑한 후 차례대로 클라이언트에 전송하는 방식이다.

  

전체 결과를 만든 후에야 전송할 수 있으므로 응답 속도가 느려진다.

  

`LIMIT`처럼 결과 건수를 제한하는 조건이 있어도 성능 향상에 큰 도움이 되지 않을 수 있다. 정렬이나 그루핑을 위해 전체 결과를 먼저 만들어야 하기 때문이다.

  

### JDBC의 버퍼링

  

다음과 같은 쿼리를 실행하면 MySQL 서버는 레코드를 읽자마자 클라이언트로 결과를 전달할 수 있다.

  

```sql

SELECT *

FROM data;

```

  

하지만 JDBC는 MySQL 서버로부터 받은 레코드를 내부 버퍼에 담아두고, 마지막 레코드까지 전달받은 뒤 클라이언트 애플리케이션에 반환할 수 있다.

  

즉 MySQL 서버는 스트리밍 방식으로 처리했더라도, 클라이언트의 JDBC 라이브러리에서 버퍼링이 발생할 수 있다.

  

## 8. GROUP BY 처리

  

`GROUP BY`도 인덱스를 사용할 수 있는 경우와 그렇지 못한 경우로 나눌 수 있다.

  

`GROUP BY`에 사용된 조건 중 `HAVING` 절은 인덱스를 사용해서 처리될 수 없다. 따라서 `HAVING` 절 자체를 튜닝하는 것은 큰 의미가 없다.

  

`GROUP BY` 처리 방식은 크게 다음과 같다.

  

- 인덱스 스캔을 이용한 `GROUP BY`

- 루스 인덱스 스캔을 이용한 `GROUP BY`

- 임시 테이블을 이용한 `GROUP BY`

  

### 인덱스 스캔을 이용한 GROUP BY

  

인덱스를 차례대로 읽으면서 그루핑 작업을 수행하고, 그 결과를 기반으로 조인을 처리한다.

  

그룹 함수를 사용하는 경우에는 임시 테이블이 필요할 때도 있다.

  

이 방식이 사용되면 실행 계획의 `Extra` 컬럼에 별도 메시지가 표시되지 않는다.

실습으로 확인해보자. `GROUP BY` 컬럼 순서가 인덱스와 맞으면 인덱스를 차례대로 읽으면서 그룹핑할 수 있다.

```sql
EXPLAIN
SELECT region, age, COUNT(*)
FROM member
GROUP BY region, age;
```

확인할 것:
![](<IMAGE/Pasted image 20260518173235.png>)

- `key`가 `idx_member_region_age`인지 확인한다.
- `Extra`에 `Using temporary`가 없다면 임시 테이블 없이 그룹핑한 것이다.

  

### 루스 인덱스 스캔을 이용한 GROUP BY

  

루스 인덱스 스캔은 인덱스의 레코드를 모두 읽지 않고, 필요한 부분만 건너뛰며 읽는 방식이다.

일반적인 인덱스 스캔은 조건에 맞는 인덱스 범위를 처음부터 끝까지 차례대로 읽는다. 이런 방식은 촘촘하게 읽는다는 의미로 **타이트 인덱스 스캔(Tight Index Scan)** 이라고 볼 수 있다.

반면 루스 인덱스 스캔은 그룹별로 필요한 첫 번째 값이나 마지막 값만 읽고, 다음 그룹으로 건너뛴다. 그래서 전체 인덱스를 다 읽지 않아도 된다.

예를 들어 다음과 같은 테이블과 인덱스가 있다고 하자.

```sql
	CREATE TABLE salaries (
	    emp_no INT NOT NULL,
	    from_date DATE NOT NULL,
	    salary INT NOT NULL,
	    PRIMARY KEY (emp_no, from_date)
	);
```

`PRIMARY KEY (emp_no, from_date)` 인덱스는 다음 순서로 정렬되어 있다.

```text
emp_no -> from_date
```

이 상태에서 다음 쿼리를 실행한다고 해보자.

```sql
EXPLAIN
SELECT emp_no
FROM salaries
WHERE from_date = '1985-03-01'
GROUP BY emp_no;
```

실행 계획의 `Extra`에 다음과 비슷한 메시지가 표시될 수 있다.
![](<IMAGE/Pasted image 20260518173323.png>)

```text
Using where; Using index for group-by
```

여기서 `Using index for group-by`가 루스 인덱스 스캔을 의미한다.

이 쿼리는 `emp_no`별로 그룹을 만들면 된다. 인덱스가 이미 `emp_no` 순서로 정렬되어 있으므로 MySQL은 같은 `emp_no` 값을 가진 인덱스 레코드를 전부 읽지 않고, 각 `emp_no` 그룹의 필요한 위치만 확인한 뒤 다음 `emp_no` 그룹으로 넘어갈 수 있다.

개념적으로는 다음과 비슷하다.

```text
emp_no = 10001 그룹에서 조건에 맞는 from_date 확인
-> emp_no = 10002 그룹으로 점프
-> emp_no = 10003 그룹으로 점프
-> ...
```

즉, 인덱스를 한 줄씩 끝까지 훑는 것이 아니라 그룹 단위로 건너뛰며 읽는다.

루스 인덱스 스캔이 특히 유리한 대표적인 경우는 그룹별 최솟값이나 최댓값을 구할 때다.

```sql
EXPLAIN
SELECT emp_no, MIN(from_date)
FROM salaries
GROUP BY emp_no;
```

`PRIMARY KEY (emp_no, from_date)` 인덱스에서는 같은 `emp_no` 안에서 `from_date`가 정렬되어 있다. 따라서 각 `emp_no` 그룹의 첫 번째 `from_date`만 읽으면 `MIN(from_date)`를 구할 수 있다.

반대로 최댓값은 각 그룹의 마지막 값만 읽으면 된다.

```sql
EXPLAIN
SELECT emp_no, MAX(from_date)
FROM salaries
GROUP BY emp_no;
```

루스 인덱스 스캔을 사용할 수 있는 대표적인 쿼리 형태는 다음과 같다. 아래 예시는 `tb_test` 테이블에 `(col1, col2, col3)` 순서의 인덱스가 있다고 가정한다.

```sql
CREATE TABLE tb_test (
    col1 INT NOT NULL,
    col2 INT NOT NULL,
    col3 INT NOT NULL,
    col4 VARCHAR(50),
    INDEX idx_col1_col2_col3 (col1, col2, col3)
);
```

인덱스의 앞쪽 컬럼 순서대로 그룹핑하면 루스 인덱스 스캔을 사용할 가능성이 있다.

```sql
EXPLAIN
SELECT col1, col2
FROM tb_test
GROUP BY col1, col2;
```

![](<IMAGE/Pasted image 20260518173450.png>)
`DISTINCT`도 내부적으로는 중복 제거가 필요하므로, 인덱스 순서와 맞으면 비슷한 방식으로 처리될 수 있다.

```sql
EXPLAIN
SELECT DISTINCT col1, col2
FROM tb_test;
```

그룹별 최솟값이나 최댓값을 구하는 경우도 루스 인덱스 스캔의 좋은 예시다.

```sql
EXPLAIN
SELECT col1, MIN(col2)
FROM tb_test
GROUP BY col1;
```

다음처럼 `WHERE` 조건이 있어도 인덱스 순서와 그룹핑 조건이 맞으면 루스 인덱스 스캔을 사용할 수 있다.

```sql
EXPLAIN
SELECT col1, col2
FROM tb_test
WHERE col1 < 10
GROUP BY col1, col2;
```

또는 그룹 기준 컬럼 뒤쪽의 컬럼에 대해 `MIN()`이나 `MAX()`를 구하는 형태도 가능하다.

```sql
EXPLAIN
SELECT col1, col2, MIN(col3), MAX(col3)
FROM tb_test
WHERE col1 < 10
GROUP BY col1, col2;
```

다만 루스 인덱스 스캔은 아무 `GROUP BY`에서나 사용할 수 있는 것은 아니다. 인덱스의 왼쪽부터 이어지는 컬럼 순서와 `GROUP BY` 컬럼 순서가 맞아야 하고, 집계 함수도 주로 `MIN()` 또는 `MAX()`처럼 인덱스의 첫 값이나 마지막 값만 읽어도 되는 형태일 때 효과적이다.

예를 들어 다음 쿼리는 인덱스가 `(col1, col2, col3)`이어도 `GROUP BY`가 인덱스의 왼쪽부터 이어지는 형태가 아니므로 루스 인덱스 스캔을 사용하기 어렵다.

```sql
EXPLAIN
SELECT col2
FROM tb_test
GROUP BY col2;
```

또한 `SUM()`, `AVG()`, `COUNT()`처럼 그룹 안의 모든 값을 확인해야 하는 집계는 그룹의 첫 값이나 마지막 값만 읽어서 계산할 수 없다. 이런 경우에는 루스 인덱스 스캔보다 일반 인덱스 스캔이나 임시 테이블을 사용하는 방식으로 처리될 가능성이 높다.

```sql
EXPLAIN
SELECT col1, SUM(col3)
FROM tb_test
GROUP BY col1;
```

정리하면, 루스 인덱스 스캔은 `GROUP BY`를 위해 인덱스를 전부 읽는 것이 아니라 **그룹별 대표 지점만 읽고 다음 그룹으로 건너뛰는 최적화**다. 그래서 그룹 수에 비해 전체 레코드 수가 많을수록 효과가 커진다.

  

이 방식이 사용되면 실행 계획의 `Extra` 컬럼에 다음 메시지가 표시된다.

  

```text

Using index for group-by

```

실습으로도 확인해보자.

```sql
EXPLAIN
SELECT member_id, MIN(reservation_time)
FROM reservation
GROUP BY member_id;
```

확인할 것:
![](<IMAGE/Pasted image 20260518173552.png>)

- `idx_reservation_member_time (member_id, reservation_time)` 인덱스를 사용할 수 있다.
- `Extra`에 `Using index for group-by`가 표시될 수 있다.

  

### 임시 테이블을 이용한 GROUP BY

  

인덱스를 전혀 사용하지 못하는 경우 임시 테이블을 이용해서 그루핑한다.

  

그룹 기준 컬럼이 드라이빙 테이블에 있든 드리븐 테이블에 있든 관계없이, 인덱스를 사용하지 못하면 이 방식으로 처리될 수 있다.

  

이 방식이 사용되면 실행 계획의 `Extra` 컬럼에 다음 메시지가 표시된다.

  

```text

Using temporary

```

실습으로 확인해보자. 인덱스 순서와 맞지 않는 컬럼으로 그룹핑하면 임시 테이블이 필요할 수 있다.

```sql
EXPLAIN
SELECT status, theme_id, COUNT(*)
FROM reservation
GROUP BY status, theme_id;
```

확인할 것:

- `Extra`에 `Using temporary`가 표시될 수 있다.

  ![](<IMAGE/Pasted image 20260518173655.png>)

MySQL 8.0부터는 `GROUP BY` 컬럼 기준으로 묵시적 정렬을 수행하지 않는다. 정렬이 필요하다면 `ORDER BY`를 명시해야 한다.

  

## 9. DISTINCT 처리

  

특정 컬럼의 유니크한 값만 조회하려면 `SELECT` 쿼리에 `DISTINCT`를 사용한다.

  

`DISTINCT`는 집합 함수와 함께 사용되는 경우와 그렇지 않은 경우를 구분해서 이해해야 한다.

  

### 단순 DISTINCT

  

단순히 유니크한 레코드만 가져오는 `SELECT DISTINCT`는 `GROUP BY`와 유사한 방식으로 처리된다.

  

```sql

SELECT DISTINCT seq_member

FROM member;

```

  

위 쿼리는 다음 쿼리와 유사하게 처리된다.

  

```sql

SELECT seq_member

FROM member

GROUP BY seq_member;

```

실습용 테이블에서는 다음 두 쿼리의 실행 계획을 비교해볼 수 있다.

```sql
EXPLAIN
SELECT DISTINCT region
FROM member;
```
![](<IMAGE/Pasted image 20260518173727.png>)
```sql
EXPLAIN
SELECT region
FROM member
GROUP BY region;
```

  
![](<IMAGE/Pasted image 20260518173756.png>)

똑같다다

주의할 점은 `DISTINCT`가 특정 컬럼 하나에만 적용되는 것이 아니라, SELECT 절에 명시된 전체 컬럼 조합에 적용된다는 것이다.

  

```sql

SELECT DISTINCT name, age

FROM member;

```

  

위 쿼리는 `name`만 유니크하게 가져오는 것이 아니라, `(name, age)` 조합이 유니크한 레코드를 가져온다.

  

다음처럼 작성해도 결과는 동일하다. MySQL은 `DISTINCT(name)`의 괄호를 특별하게 해석하지 않는다.

  

```sql

SELECT DISTINCT(name), age

FROM member;

```

실제로 다음 쿼리는 `region`만 유니크하게 만드는 것이 아니라 `(region, age)` 조합이 유니크한 결과를 반환한다.

```sql
SELECT DISTINCT region, age
FROM member
ORDER BY region, age
LIMIT 20;
```

  

### 집합 함수와 함께 사용된 DISTINCT

  

`COUNT()`, `MIN()`, `MAX()` 같은 집합 함수 안에서도 `DISTINCT`를 사용할 수 있다.

  

이때의 `DISTINCT`는 집합 함수의 인자로 전달된 컬럼값 중 유니크한 값만 대상으로 집계한다.

  

```sql

SELECT COUNT(DISTINCT t.name)

FROM member m, tmp t

WHERE t.seq_member = m.seq_member

  AND m.seq_member BETWEEN 1001 AND 1005;

```

  

위 쿼리는 `COUNT(DISTINCT t.name)`을 처리하기 위해 내부적으로 임시 테이블을 생성할 수 있다.

실습용 테이블에서는 다음처럼 확인해볼 수 있다.

```sql
EXPLAIN
SELECT COUNT(DISTINCT region)
FROM member;
```

  ![](<IMAGE/Pasted image 20260518173854.png>)

만약 서로 다른 컬럼에 대해 각각 `COUNT(DISTINCT ...)`를 수행한다면, 필요한 임시 테이블도 각각 생성될 수 있다.

  

```sql

SELECT COUNT(DISTINCT t.name),

       COUNT(DISTINCT m.name)

FROM member m, tmp t

WHERE t.seq_member = m.seq_member

  AND m.seq_member BETWEEN 1001 AND 1005;

```

  

위 쿼리에서는 `COUNT(DISTINCT t.name)`과 `COUNT(DISTINCT m.name)`을 처리하기 위해 내부적으로 별도의 임시 테이블이 필요할 수 있다.

서로 다른 테이블의 컬럼을 각각 `COUNT(DISTINCT ...)`로 집계하는 실습은 다음처럼 해볼 수 있다.

```sql
EXPLAIN
SELECT COUNT(DISTINCT m.region),
       COUNT(DISTINCT r.status)
FROM member m
JOIN reservation r ON r.member_id = m.id;
```

확인할 것:

- `COUNT(DISTINCT ...)`는 중복 제거를 위한 내부 처리가 필요하다.
- 서로 다른 DISTINCT 대상이 여러 개면 각각 별도의 중복 제거 작업이 필요할 수 있다.


## 10. 직접 해보는 퀴즈

아래 퀴즈는 앞에서 만든 `optimizer_lab` 데이터베이스를 그대로 사용한다. 먼저 답을 예상한 뒤 `EXPLAIN` 또는 `EXPLAIN ANALYZE`로 실제 실행 계획을 확인해보자.

### 퀴즈 1. 풀 테이블 스캔과 풀 인덱스 스캔 구분하기

다음 두 쿼리의 실행 계획을 비교해보자.

```sql
EXPLAIN
SELECT *
FROM member
WHERE bio LIKE '%100%';

EXPLAIN
SELECT COUNT(*)
FROM member;
```

질문:

1. 각 쿼리의 `type`은 무엇인가?
2. 두 쿼리 모두 테이블 전체를 훑는 것처럼 보이는데, 왜 하나는 풀 테이블 스캔이고 다른 하나는 풀 인덱스 스캔일 수 있을까?
3. `SELECT COUNT(*)`가 `SELECT *`보다 더 작은 데이터를 읽을 수 있는 이유는 무엇일까?

확인 포인트:

- `bio LIKE '%100%'`는 앞쪽 와일드카드 때문에 일반 B-Tree 인덱스를 사용하기 어렵다.
- `COUNT(*)`는 실제 레코드 전체 컬럼을 읽지 않아도 되므로, 더 작은 인덱스를 읽는 계획이 선택될 수 있다.

### 퀴즈 2. 인덱스를 이용한 정렬 찾기

다음 세 쿼리 중 `Using filesort`가 발생하지 않을 가능성이 가장 높은 쿼리를 골라보자.

```sql
EXPLAIN
SELECT id, age, joined_at
FROM member
ORDER BY age, joined_at
LIMIT 10;

EXPLAIN
SELECT id, age, joined_at
FROM member
ORDER BY joined_at, age
LIMIT 10;

EXPLAIN
SELECT id, age, joined_at
FROM member
ORDER BY name
LIMIT 10;
```

질문:

1. 어떤 쿼리가 `idx_member_age_joined_at` 인덱스 순서를 그대로 사용할 수 있는가?
2. `(age, joined_at)` 인덱스가 있는데도 `ORDER BY joined_at, age`는 왜 불리할까?
3. `LIMIT 10`이 있어도 Filesort가 필요하면 왜 비용이 커질 수 있을까?

확인 포인트:

- 복합 인덱스는 왼쪽 컬럼부터 정렬되어 있다.
- 인덱스 순서와 `ORDER BY` 순서가 맞으면 이미 정렬된 인덱스를 앞에서부터 읽을 수 있다.
- Filesort는 결과 10건만 반환하더라도, 어떤 10건이 앞에 올지 판단하기 위해 더 많은 레코드를 정렬해야 할 수 있다.

### 퀴즈 3. 정렬 상태 변수로 Filesort 확인하기

다음 순서대로 실행해보자.

```sql
SHOW SESSION STATUS LIKE 'Sort%';

SELECT id, age, joined_at
FROM member
ORDER BY name
LIMIT 10;

SHOW SESSION STATUS LIKE 'Sort%';
```

이어서 인덱스를 사용할 수 있는 정렬도 실행해보자.

```sql
SHOW SESSION STATUS LIKE 'Sort%';

SELECT id, age, joined_at
FROM member
ORDER BY age, joined_at
LIMIT 10;

SHOW SESSION STATUS LIKE 'Sort%';
```

질문:

1. `ORDER BY name` 실행 후 `Sort_rows` 값은 증가했는가?
2. `ORDER BY age, joined_at` 실행 후에도 같은 방식으로 증가하는가?
3. 두 결과가 다르다면, 그 이유를 실행 계획의 `Extra`와 연결해서 설명해보자.

확인 포인트:

- `Sort_rows`는 실제 정렬된 레코드 수를 보여준다.
- 인덱스를 이용한 정렬은 별도 Filesort가 필요 없으므로 정렬 관련 상태 변수 증가가 없거나 훨씬 적을 수 있다.

### 퀴즈 4. 조인에서 정렬 대상 줄이기

다음 두 쿼리의 실행 계획을 비교해보자.

```sql
EXPLAIN
SELECT m.id, m.name, r.id AS reservation_id
FROM member m
STRAIGHT_JOIN reservation r ON r.member_id = m.id
ORDER BY m.name
LIMIT 10;

EXPLAIN
SELECT m.id, m.name, r.id AS reservation_id, r.reservation_time
FROM member m
STRAIGHT_JOIN reservation r ON r.member_id = m.id
ORDER BY r.reservation_time
LIMIT 10;
```

질문:

1. 두 쿼리의 `ORDER BY` 기준 컬럼은 각각 어느 테이블에 있는가?
2. 첫 번째 쿼리는 왜 드라이빙 테이블만 정렬한 뒤 조인을 수행할 수 있을까?
3. 두 번째 쿼리는 왜 `Using temporary; Using filesort`가 발생할 가능성이 높을까?

확인 포인트:

- `STRAIGHT_JOIN`을 사용하면 `member`를 먼저 읽도록 고정할 수 있다.
- 드라이빙 테이블 컬럼으로 정렬하면 조인 전에 정렬 대상을 줄일 수 있다.
- 드리븐 테이블 컬럼으로 정렬하면 조인 결과를 만든 뒤 정렬해야 하므로 임시 테이블이 필요할 수 있다.

### 퀴즈 5. GROUP BY 처리 방식 맞히기

다음 세 쿼리의 `Extra`를 예상한 뒤 확인해보자.

```sql
EXPLAIN
SELECT region, age, COUNT(*)
FROM member
GROUP BY region, age;

EXPLAIN
SELECT member_id, MIN(reservation_time)
FROM reservation
GROUP BY member_id;

EXPLAIN
SELECT status, theme_id, COUNT(*)
FROM reservation
GROUP BY status, theme_id;
```

질문:

1. 어떤 쿼리가 인덱스 스캔을 이용한 `GROUP BY`로 처리될 가능성이 높은가?
2. 어떤 쿼리에서 `Using index for group-by`가 표시될 수 있는가?
3. 어떤 쿼리에서 `Using temporary`가 표시될 가능성이 높은가?

확인 포인트:

- `member`에는 `(region, age)` 인덱스가 있다.
- `reservation`에는 `(member_id, reservation_time)` 인덱스가 있다.
- `status, theme_id` 순서의 인덱스는 없으므로 임시 테이블을 사용할 가능성이 높다.

### 퀴즈 6. DISTINCT의 범위 확인하기

다음 두 쿼리의 결과 건수를 비교해보자.

```sql
SELECT COUNT(*) AS distinct_region_count
FROM (
    SELECT DISTINCT region
    FROM member
) t;

SELECT COUNT(*) AS distinct_region_age_count
FROM (
    SELECT DISTINCT region, age
    FROM member
) t;
```

질문:

1. 두 결과 건수는 같은가, 다른가?
2. `SELECT DISTINCT region, age`는 `region`만 중복 제거하는 쿼리인가?
3. `DISTINCT(region), age`처럼 괄호를 사용하면 의미가 달라질까?

확인 포인트:

- `DISTINCT`는 특정 컬럼 하나가 아니라 SELECT 절의 전체 컬럼 조합에 적용된다.
- `(region)`처럼 괄호를 써도 MySQL이 `region` 하나에만 `DISTINCT`를 적용하는 것은 아니다.

### 퀴즈 7. 인덱스 하나 추가해서 실행 계획 바꿔보기

먼저 아래 쿼리의 실행 계획을 확인한다.

```sql
EXPLAIN
SELECT status, theme_id, COUNT(*)
FROM reservation
GROUP BY status, theme_id;
```

그 다음 인덱스를 추가하고 다시 확인한다.

```sql
CREATE INDEX idx_reservation_status_theme
ON reservation (status, theme_id);

EXPLAIN
SELECT status, theme_id, COUNT(*)
FROM reservation
GROUP BY status, theme_id;
```

질문:

1. 인덱스 추가 전후로 `key`가 바뀌었는가?
2. `Extra`에서 `Using temporary`가 사라졌는가?
3. 이 인덱스는 어떤 조회에는 도움이 되지만, 어떤 비용을 추가로 만들까?

확인 포인트:

- `GROUP BY status, theme_id`와 같은 순서의 인덱스를 만들면 그룹핑에 사용할 수 있다.
- 인덱스는 읽기 성능을 높일 수 있지만, INSERT/UPDATE/DELETE 시 인덱스 유지 비용과 저장 공간 비용이 추가된다.

실습 후 원래 상태로 되돌리고 싶다면 다음 명령을 실행하면 된다.

```sql
DROP INDEX idx_reservation_status_theme ON reservation;
```

### 퀴즈 8. 내가 옵티마이저라면?

다음 요구사항을 만족하는 쿼리를 직접 작성하고, 실행 계획을 확인해보자.

요구사항:

- `SEOUL` 지역 회원만 조회한다.
- 나이가 어린 순서, 가입일이 빠른 순서로 정렬한다.
- 상위 10명만 가져온다.

예상 쿼리:

```sql
EXPLAIN
SELECT id, name, region, age, joined_at
FROM member
WHERE region = 'SEOUL'
ORDER BY age, joined_at
LIMIT 10;
```

질문:

1. 이 쿼리는 어떤 인덱스를 사용할 가능성이 높은가?
2. `idx_member_region_age (region, age)`만으로 `ORDER BY age, joined_at`까지 완전히 해결할 수 있을까?
3. 이 쿼리를 더 정렬 친화적으로 만들려면 어떤 복합 인덱스를 고려할 수 있을까?

확인 포인트:

- `WHERE region = 'SEOUL'` 조건에는 `(region, age)` 인덱스가 도움이 될 수 있다.
- 하지만 `joined_at`까지 포함해 정렬을 완전히 맞추려면 `(region, age, joined_at)` 같은 인덱스가 더 적합할 수 있다.
- 인덱스를 추가하기 전에는 항상 실제 조회 패턴과 쓰기 비용을 함께 고려해야 한다.

### 퀴즈 정답 및 해설

#### 퀴즈 1 정답

첫 번째 쿼리는 풀 테이블 스캔이 발생할 가능성이 높다.

```sql
SELECT *
FROM member
WHERE bio LIKE '%100%';
```

`bio LIKE '%100%'`는 검색어 앞에 `%`가 붙어 있어서 B-Tree 인덱스를 이용해 시작 지점을 찾기 어렵다. 그래서 `type`이 `ALL`로 표시될 수 있다. `Extra`에는 읽어온 레코드를 `WHERE` 조건으로 다시 필터링했다는 의미로 `Using where`가 표시될 수 있다.

두 번째 쿼리는 풀 인덱스 스캔이 발생할 가능성이 높다.

```sql
SELECT COUNT(*)
FROM member;
```

전체 건수를 세는 쿼리는 모든 컬럼 값을 읽을 필요가 없다. MySQL은 테이블 전체 레코드보다 크기가 작은 인덱스를 처음부터 끝까지 읽어서 건수를 셀 수 있다. 이 경우 `type`이 `index`로 표시될 수 있다.

정리:

- `type = ALL`: 풀 테이블 스캔
- `type = index`: 풀 인덱스 스캔
- `Using where`: 읽은 레코드에 대해 `WHERE` 조건을 적용했다는 뜻

#### 퀴즈 2 정답

`Using filesort`가 발생하지 않을 가능성이 가장 높은 쿼리는 첫 번째 쿼리다.

```sql
SELECT id, age, joined_at
FROM member
ORDER BY age, joined_at
LIMIT 10;
```

`member` 테이블에는 다음 인덱스가 있다.

```sql
INDEX idx_member_age_joined_at (age, joined_at)
```

인덱스가 `(age, joined_at)` 순서로 정렬되어 있으므로 `ORDER BY age, joined_at`은 인덱스 순서를 그대로 사용할 수 있다.

반면 다음 쿼리는 인덱스 컬럼 순서와 `ORDER BY` 순서가 다르다.

```sql
ORDER BY joined_at, age
```

`(age, joined_at)` 인덱스는 먼저 `age`로 정렬되고, 같은 `age` 안에서 `joined_at`으로 정렬된다. 따라서 전체 결과를 `joined_at` 먼저 정렬하는 데 그대로 사용할 수 없다.

`ORDER BY name`도 `name`에 맞는 인덱스가 없으므로 Filesort가 발생할 가능성이 높다.

#### 퀴즈 3 정답

`ORDER BY name` 쿼리는 Filesort가 발생할 가능성이 높으므로 `Sort_rows`가 증가할 수 있다.

```sql
SELECT id, age, joined_at
FROM member
ORDER BY name
LIMIT 10;
```

`name` 컬럼에는 정렬에 사용할 수 있는 인덱스가 없기 때문이다. 실행 계획의 `Extra`에 `Using filesort`가 표시될 가능성이 높다.

반면 다음 쿼리는 `(age, joined_at)` 인덱스를 사용할 수 있다.

```sql
SELECT id, age, joined_at
FROM member
ORDER BY age, joined_at
LIMIT 10;
```

따라서 별도의 Filesort 없이 인덱스 순서대로 읽을 수 있고, 정렬 관련 상태 변수 증가가 없거나 훨씬 적을 수 있다.

정리:

- `ORDER BY name`: Filesort 가능성 높음, `Sort_rows` 증가 가능
- `ORDER BY age, joined_at`: 인덱스 정렬 가능성 높음, Filesort 관련 상태 변수 증가가 적음

#### 퀴즈 4 정답

첫 번째 쿼리는 정렬 기준이 드라이빙 테이블인 `member`에 있다.

```sql
ORDER BY m.name
```

`STRAIGHT_JOIN`을 사용했으므로 MySQL은 `member`를 먼저 읽는다. 따라서 `member` 레코드만 먼저 `m.name` 기준으로 정렬한 뒤, 정렬된 순서대로 `reservation`을 조인할 수 있다.

이 경우 `Extra`에는 `Using filesort`가 표시될 수 있지만, `Using temporary`는 없을 수 있다. 조인 결과 전체를 임시 테이블에 담아 정렬하는 방식은 아니기 때문이다.

두 번째 쿼리는 정렬 기준이 드리븐 테이블인 `reservation`에 있다.

```sql
ORDER BY r.reservation_time
```

`member`를 먼저 읽는 상황에서 `reservation`의 값으로 최종 정렬해야 하므로, `member`만 먼저 정렬해서는 전체 결과 순서를 보장할 수 없다. 그래서 조인 결과를 만든 뒤 임시 테이블에 담고 정렬해야 할 가능성이 높다.

정리:

- `ORDER BY m.name`: 드라이빙 테이블만 정렬 가능
- `ORDER BY r.reservation_time`: 조인 결과를 임시 테이블에 저장한 뒤 정렬할 가능성 높음

#### 퀴즈 5 정답

첫 번째 쿼리는 인덱스 스캔을 이용한 `GROUP BY`로 처리될 가능성이 높다.

```sql
SELECT region, age, COUNT(*)
FROM member
GROUP BY region, age;
```

`member` 테이블에는 `(region, age)` 인덱스가 있으므로, 인덱스 순서대로 읽으면서 그룹핑할 수 있다. `Extra`에 `Using temporary`가 없다면 임시 테이블 없이 처리된 것이다.

두 번째 쿼리는 루스 인덱스 스캔을 사용할 가능성이 있다.

```sql
SELECT member_id, MIN(reservation_time)
FROM reservation
GROUP BY member_id;
```

`reservation` 테이블에는 `(member_id, reservation_time)` 인덱스가 있다. 같은 `member_id` 안에서 `reservation_time`이 정렬되어 있으므로, 각 회원별 첫 번째 `reservation_time`만 확인하면 `MIN(reservation_time)`을 구할 수 있다. 이때 `Extra`에 `Using index for group-by`가 표시될 수 있다.

세 번째 쿼리는 임시 테이블을 사용할 가능성이 높다.

```sql
SELECT status, theme_id, COUNT(*)
FROM reservation
GROUP BY status, theme_id;
```

`reservation` 테이블에는 `(status, theme_id)` 순서의 인덱스가 없기 때문이다. 따라서 `Extra`에 `Using temporary`가 표시될 수 있다.

#### 퀴즈 6 정답

두 결과 건수는 다를 가능성이 높다.

```sql
SELECT DISTINCT region
FROM member;
```

이 쿼리는 `region` 값만 유니크하게 만든다. 실습 데이터에서는 지역이 5개이므로 결과도 5건이 된다.

```sql
SELECT DISTINCT region, age
FROM member;
```

이 쿼리는 `region`만 중복 제거하는 것이 아니라 `(region, age)` 조합을 유니크하게 만든다. 따라서 지역 5개와 나이 여러 개의 조합만큼 결과가 늘어날 수 있다.

또한 다음처럼 괄호를 써도 의미는 달라지지 않는다.

```sql
SELECT DISTINCT(region), age
FROM member;
```

MySQL은 `DISTINCT(region)`을 `region` 컬럼 하나에만 적용하는 특수 문법으로 해석하지 않는다. `DISTINCT`는 SELECT 절 전체 컬럼 조합에 적용된다.

#### 퀴즈 7 정답

인덱스 추가 전에는 다음 쿼리에서 `Using temporary`가 표시될 가능성이 높다.

```sql
SELECT status, theme_id, COUNT(*)
FROM reservation
GROUP BY status, theme_id;
```

기존에는 `(status, theme_id)` 순서의 인덱스가 없기 때문이다.

다음 인덱스를 추가하면 실행 계획이 바뀔 수 있다.

```sql
CREATE INDEX idx_reservation_status_theme
ON reservation (status, theme_id);
```

이후 같은 쿼리를 다시 실행하면 `key`에 `idx_reservation_status_theme`이 표시될 수 있고, `Extra`에서 `Using temporary`가 사라질 수 있다.

하지만 이 인덱스가 항상 좋은 것은 아니다. 인덱스가 추가되면 `reservation` 테이블에 데이터를 추가하거나 수정하거나 삭제할 때 해당 인덱스도 함께 갱신해야 한다. 또한 디스크 공간도 더 사용한다.

정리:

- 장점: `GROUP BY status, theme_id` 같은 조회에 유리
- 단점: 쓰기 성능 저하 가능, 저장 공간 증가

#### 퀴즈 8 정답

요구사항을 만족하는 쿼리는 다음처럼 작성할 수 있다.

```sql
SELECT id, name, region, age, joined_at
FROM member
WHERE region = 'SEOUL'
ORDER BY age, joined_at
LIMIT 10;
```

현재 인덱스 중에서는 다음 인덱스를 사용할 가능성이 높다.

```sql
INDEX idx_member_region_age (region, age)
```

`WHERE region = 'SEOUL'` 조건으로 먼저 범위를 줄이고, 그 안에서 `age` 순서로 읽을 수 있기 때문이다.

하지만 이 인덱스는 `(region, age)`까지만 포함한다. `ORDER BY age, joined_at`에서 `joined_at`까지 완전히 정렬하려면 부족할 수 있다. 같은 `region`, 같은 `age` 안에서 `joined_at` 순서까지 보장하려면 다음과 같은 인덱스가 더 적합하다.

```sql
CREATE INDEX idx_member_region_age_joined_at
ON member (region, age, joined_at);
```

이 인덱스가 있으면 `region = 'SEOUL'` 조건으로 특정 지역만 좁힌 뒤, 그 범위 안에서 `age, joined_at` 순서대로 읽을 수 있다.

다만 실제 서비스에서는 이 인덱스를 무조건 추가하기보다, 해당 조회가 자주 실행되는지와 쓰기 비용이 늘어나는 것을 감수할 만한지 함께 판단해야 한다.
